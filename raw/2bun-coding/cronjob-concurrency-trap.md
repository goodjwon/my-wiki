## 영상 요약

크론 스케줄러 사용 시 발생할 수 있는 중복 실행 문제와 이를 방어하기 위한 인프라 설정.

### 1. 현상: 월말 정산 금액이 두 배

- 월말 정산 배치 후 정산 금액이 정확히 두 배로 불어나는 장애 발생.
- 코드에는 버그 없음. 원인: **이전 배치가 끝나기 전 다음 스케줄 시간이 도래 → 배치 동시 두 번 실행**. [00:00]

### 2. 크론잡의 기본 동작과 중복 실행 위험

- **Linux Cron / Kubernetes CronJob**: 기본적으로 이전 작업 완료 여부 확인 없이 정해진 시간에 무조건 새 프로세스 띄움. [00:10]
- 쿠버네티스 CronJob의 동시성 정책(`concurrencyPolicy`) 기본값 = **`Allow`**: 이전 작업 돌고 있어도 새 작업 무조건 허용. [00:20]
- 두 배치가 동시에 같은 DB 테이블 `INSERT` 또는 파일 시스템 조작 → 데이터 중복·파일 꼬임 같은 치명적 장애. [00:33]

### 3. 해결 및 방어 설계

- **Linux**: `flock`(파일 락) 명령어 조합으로 동일 작업 중복 실행 방지. [00:43]
- **Kubernetes**: 동시성 정책을 **`Forbid`** 로 변경 → 이전 작업이 러닝 중이면 다음 스케줄 스킵. [00:43]

### 4. `Forbid` 의 숨겨진 함정과 최종 해결책

- 단순 `Forbid`만으로는 부족.
- 이전 배치가 네트워크 타임아웃·데드락 등으로 **Hang 상태에 빠지면 → 다음 스케줄이 영원히 스킵** → 배치 자체가 안 돔. [00:53]
- 실무에서는 `Forbid` + **`activeDeadlineSeconds`** 옵션 필수. [01:03]
- 지정 시간 초과 시 Hang 걸린 배치를 **강제 종료(Kill)** → 최소한 다음 주기 스케줄은 정상 실행 보장.

### 요약

크론잡의 기본 설정은 언제나 중복 실행을 허용. 이를 단순 차단하면 Hang에 취약. **동시성 제한(`Forbid`) + 강제 종료 타임아웃(`activeDeadlineSeconds`)** 을 함께 걸어주는 **방어적 인프라 설계**가 필수. [01:16]
