## 영상 요약

DB 스키마에서 자주 쓰이는 `VARCHAR(255)`의 동작 원리와 문자셋 변경에 따른 최적화 기준.

### 1. `VARCHAR(255)`의 유래 — 1바이트 프리픽스 한계

- MySQL InnoDB는 `VARCHAR` 저장 시 문자열 앞에 **길이 프리픽스(Length Prefix)** 를 붙임. [00:24]
- 프리픽스 1바이트로 표현 가능한 숫자 범위 = `0~255` → 과거부터 255가 관습. [00:24]

### 2. 문자셋 변화에 따른 경계값 변화

- `VARCHAR(N)`의 N은 **'문자 수(글자 수)'** 이지 바이트 수가 아님. [00:36]
- 과거 **Latin1** 시절: 1글자 = 1바이트 → `VARCHAR(255)` = 정확히 1바이트 프리픽스 영역(≤ 255B). [01:39]
- 현대 **utf8mb4**: 1글자 최대 **4바이트**. [00:36]
- utf8mb4 기준 1바이트 프리픽스 영역 = `VARCHAR(63)` (63 × 4 = 252B). [01:25]
- 관습대로 `VARCHAR(64)` 또는 `VARCHAR(255)` 지정 시 프리픽스가 **2바이트로 늘어남**. [00:36], [01:25]

### 3. 프리픽스가 늘어날 때의 성능 영향

- 1바이트 → 2바이트는 사소해 보이지만:
  - 데이터 1억 건 시 **100MB 이상의 저장 공간 낭비**. [00:50]
  - 인덱스(B-Tree)에도 프리픽스 포함 → 페이지 1개당 들어갈 키 수 감소 → **페이지 스플릿(Page Split) 잦아짐** → 쿼리 성능 저하. [00:50]

### 4. DB 제품별 차이와 결론

- 모든 DB가 같은 규칙은 아님.
- **PostgreSQL**: 'varlena' 구조 사용. MySQL 같은 255바이트 경계 없음. 255 vs 256 저장 비용 차이 없음. [01:03]
- 데이터 잘림 위험 감수하며 무조건 숫자를 맞추는 것은 본말전도. [01:39]

### 요약

`VARCHAR(255)`는 과거 Latin1 시절의 최적값. 현대 utf8mb4에서 1바이트 프리픽스 효과를 내려면 `VARCHAR(63)`이 정확. 결국 컬럼 크기는 낡은 관습보다 **저장될 실제 데이터의 도메인 특성에 맞춰 결정**. [01:25], [01:39]
