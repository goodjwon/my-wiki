## 영상 요약

트래픽이 적은 새벽 시간에 간헐적으로 발생하는 **502 Bad Gateway** 에러의 원인과 해결책. 인프라(LB)와 웹 서버 간 Keep-Alive 타임아웃 불일치로 인한 race condition.

### 1. 현상: 새벽에만 간헐적으로 발생하는 502

- 새벽에 502 에러 발생 → 서버 확인 시 CPU·메모리 모두 여유, 서버 로그도 깨끗. [00:00]
- 낮에는 절대 안 발생, 오직 요청이 드문 **새벽 시간에만** 발생. [00:10]

### 2. 원인: LB ↔ 서버의 Keep-Alive 타임아웃 불일치

- LB와 서버는 연결 비용을 줄이기 위해 **Keep-Alive** 방식으로 TCP 커넥션 유지. [00:21]
- 문제: 커넥션 유지 시간(idle timeout)을 **LB와 서버가 서로 다르게 설정**. [00:32]

**예시 (AWS ALB + 일반 웹 서버)**:
- AWS ALB 기본 유휴 타임아웃: **60초** [00:45]
- 웹 서버 기본 Keep-Alive 타임아웃: 훨씬 짧음 [00:57]
  - Gunicorn: **2초**
  - Node.js: **5초**

**Race condition 시나리오 [01:08]**:
- 트래픽 없는 새벽, 서버 타임아웃(2초)에 도달해 커넥션 끊으려는 순간
- LB는 아직 60초가 안 지나 커넥션 살아있다고 판단 → 새 요청 전송
- 서버가 끊는 순간 ↔ LB가 보내는 순간 겹침
- LB가 유효하지 않은 커넥션에 요청 → 502 반환

- 낮에는 요청이 끊임없어서 타임아웃 전에 커넥션 재사용 → 문제 안 생김. [01:30]

### 3. 해결: 서버 타임아웃 > LB 타임아웃

- **웹 서버 Keep-Alive 타임아웃을 LB 타임아웃보다 더 길게** 설정. [01:41]
- LB가 항상 먼저 끊게 만들면, 서버가 이미 종료한 커넥션으로 LB가 요청 보내는 구조적 결함 사라짐.
- AWS 공식 문서 권장 사항.

### 4. 주의: ALB → NLB 전환 시

- ALB(L7)에서 NLB(L4)로 전환 시 유휴 타임아웃이 **350초**로 늘어남. [01:52]
- 서버 타임아웃을 ALB 기준(예: 65초)으로 설정했다면, NLB 전환 후 NLB(350초) > 서버(65초)가 되어 **동일한 502 재발생**. [02:04]

### 요약

간헐적 502의 원인은 서버 애플리케이션 코드가 아니라 **인프라(LB)와 웹 서버 사이의 타이머 불일치**. 서버 모니터링 지표로는 절대 못 찾으므로 네트워크 인프라 설정을 함께 점검해야 함. [02:17]
