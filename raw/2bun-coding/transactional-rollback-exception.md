# @Transactional은 왜 RuntimeException에만 롤백할까요? (트랜잭션의 설계)

## 영상 요약

스프링 `@Transactional` 어노테이션이 특정 예외 상황에서 자동 롤백을 수행하지 않는 이유와 그 배경의 자바 예외 처리 철학.

### 1. 현상: 예외가 났는데 롤백이 안 됨

- 상위 레이어로 예외를 던졌음에도 DB에 INSERT/UPDATE된 데이터가 그대로 남아있는 함정. [00:00]
- 원인: 스프링 트랜잭션의 기본 동작이 **언체크 예외(Unchecked Exception / RuntimeException)에만 자동 롤백** 되도록 설계. [00:12], [00:45]

### 2. 자바의 예외 철학과 스프링의 설계 일관성

- 자바 설계자들이 두 가지 예외에 다른 의미 부여. [00:22]
- **체크 예외 (Checked)**: 복구·예측 가능한 '외부 요인 실패' (예: `IOException`, `SQLException`). [00:34]
- **언체크 예외 (Unchecked / RuntimeException)**: '코드가 잘못된 결과(프로그래밍 에러)' — 시스템 진행 불가. [00:34]
- 스프링이 이 철학을 그대로 수용:
  - 체크 예외 = 복구 가능 → **트랜잭션 살림 (롤백 안 함)**
  - 언체크 예외 = 복구 불가 → **자동 롤백**. [00:45]

### 3. 실무와의 괴리 & 해결책: `rollbackFor`

- 현대 실무에서는 `IOException`·`SQLException`이 발생해도 즉시 복구하기보다 **상위로 던지면서 트랜잭션도 롤백되길 기대**. [00:56]
- 관습적 해결:
```java
@Transactional(rollbackFor = Exception.class)
```
- 체크/언체크 가리지 않고 **모든 예외 상황에서 전부 롤백 요청**. [01:07], [01:18]
- 30년 전 자바 예외 철학 대신 현실 비즈니스 요구사항에 맞춰 재정의.

### 4. 현대 개발 트렌드 — 체크 예외에 회의적

- **Kotlin**: 체크 예외 개념 **아예 없앰**. [01:29]
- **모던 자바**: 체크 예외를 `RuntimeException`으로 감싸서(wrap) 던지는 패턴 일반화. [01:39]

### 요약

스프링 트랜잭션의 기본 롤백 동작은 1990년대 자바 설계자들의 고전적 예외 철학 반영. 어노테이션 한 줄 또는 `rollbackFor = Exception.class` 설정으로 우리는 **30년 전 결정을 매일 제어**하고 있다. [01:40], [01:52]
