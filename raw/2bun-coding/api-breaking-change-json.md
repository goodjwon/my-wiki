## 영상 요약

API 설계 시 필드를 추가하는 사소한 변경이 특정 클라이언트 환경에서 치명적 장애(Breaking Change)를 유발하는 원인과 대책.

### 1. 현상: 기존 필드 유지 + 신규 필드 추가만 했는데 특정 앱이 먹통

- 웹 프론트엔드 요구로 기존 필드 유지하고 새 필드만 응답에 추가 → 배포. [00:00]
- 결과: 웹·안드로이드 A팀은 정상, **안드로이드 B팀 앱에서만 해당 API 호출 전부 실패** 장애. [00:11]

### 2. 원인: 클라이언트 JSON 파싱 라이브러리의 전략 차이

- 같은 안드로이드라도 **JSON 역직렬화 라이브러리의 기본 설정이 다름**. [00:11]
- **안드로이드 A (Gson) & 웹 (JavaScript)**: 모르는 필드 무시, 필요한 데이터만 꺼냄 → 에러 안 남. [00:23]
- **안드로이드 B (kotlinx.serialization)**: 기본이 매우 엄격. DTO에 정의 안 된 미지 필드가 들어오면 **역직렬화 자체 실패 (Exception)**. [00:36]
- 서버 입장에선 단순 '추가'였지만, 클라이언트 라이브러리 특성 때문에 하위 호환성 깨짐. [00:48]

### 3. API 버전 관리의 한계

- 장애 후 `v1`/`v2` 분리 고려하게 되지만 만능 아님. [01:00]
- 강제 업데이트 불가 환경(예: B2B 고객사 앱) — `v1` 사용자가 앱 업데이트 안 하면 **서버는 과거 버전 코드를 영원히 유지하고 분기 처리** 해야 하는 유지보수 부담. [01:10]

### 4. 해결책: 명세와 계약(Contract)

- 서버가 클라이언트의 파싱 라이브러리 코드까지 강제할 수 없음.
- 그래서 **API 명세서에 "클라이언트는 정의되지 않은 알 수 없는 필드를 반드시 무시해야 한다"는 규칙을 명시** 하고 상호 약속. [01:20]

### 요약

API의 하위 호환성은 서버 개발자 기준이 아니라 **가장 엄격하게 구현된 클라이언트 라이브러리를 기준**으로 판단해야 한다. 무심코 추가한 필드 하나가 다른 개발자에게 전면 서비스 장애로 다가올 수 있다. [01:41]
