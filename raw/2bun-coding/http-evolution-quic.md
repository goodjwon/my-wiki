## 영상 요약

HTTP 프로토콜이 1.1 → 2 → 3으로 발전해 온 과정과 각 버전에서 해결하고자 했던 **네트워크 병목(HOL 블로킹)** 의 이동.

### 1. HTTP 1.1 — Keep-Alive와 HOL 블로킹 한계

- 초기 HTTP: 파일 1개당 TCP 3-way handshake 새로 (HTML, CSS, JS, 이미지 각각). [00:00]
- **HTTP 1.1**: `Keep-Alive` 도입 → 하나의 커넥션 재사용. [00:12]
- 한계: 한 커넥션 내에서 **요청 순서대로만 응답이 와야 함**. 앞 응답 늦으면 뒤 요청 전부 대기 = **HOL (Head-of-Line) 블로킹**. [00:24]
- 브라우저들의 우회: 서버당 동시 **6개 커넥션** 열기. [00:34]

### 2. HTTP 2 — 멀티플렉싱과 TCP 레벨의 새로운 HOL

- **HTTP 2**: 하나의 TCP 커넥션 내에서 여러 요청 동시 송신, 순서 무관하게 응답 수신 = **멀티플렉싱(Multiplexing)**. [00:46]
- 애플리케이션 레벨 HOL은 사라짐. [00:56]
- 그러나 하위 **TCP 계층**에 문제 — TCP는 신뢰성을 위해 패킷 순서 보장 → **패킷 하나 유실되면 뒤 패킷들 전부 멈추고 재전송 대기**. [00:56]
- 결국 TCP 레벨에서 HOL 블로킹 재발. [01:10]

### 3. HTTP 3 — TCP를 버리고 UDP(QUIC)로

- **HTTP 3**: 전송 계층 한계 극복 위해 **TCP를 버리고 UDP 선택**. [01:21]
- UDP 위에 **QUIC** 이라는 새 전송 계층 프로토콜. [01:21]
- QUIC는 **각 스트림별로 독립적 패킷 관리** → 특정 스트림에서 유실 발생해도 다른 스트림은 영향 없이 데이터 송수신. [01:34]

### 4. HTTP 3의 한계와 실무 적용

- HTTP 3도 만능 아님:
  1. 기업용 방화벽이 보안상 **UDP 트래픽 차단**하는 경우 많음. [01:44]
  2. QUIC은 커널 레벨이 아닌 **애플리케이션 레벨**에서 동작 → 기존 TCP보다 **CPU 사용량이 더 높음**. [01:44]
- 실무: 대부분의 서버가 **HTTP 2 + HTTP 3 동시 활성화**, 클라이언트 환경에 따라 선택. [01:57]

### 요약

HTTP의 발전 역사는 **HOL 블로킹이라는 같은 문제를 해결하기 위해 네트워크 계층을 한 단계씩 내려가며 최적화한 기록**. 애플리케이션 레벨(HTTP 2)에서 해결 → 그래도 안 되자 결국 전송 계층을 TCP → UDP/QUIC(HTTP 3)으로 교체. [01:57], [02:09]
