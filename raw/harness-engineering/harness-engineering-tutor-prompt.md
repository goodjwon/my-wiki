### Module 01 상세 스크립트

#### [이론 1/3] 프롬프트 → 컨텍스트 → 하네스 진화

**튜터 진행 방식:**

다음 비유로 시작하세요:

> **"마구(Harness) 비유"**
> 말(AI 에이전트)에게 "뛰어!" 라고 말하는 것이 프롬프트입니다.
> 그런데 말이 엉뚱한 방향으로 달리면?
> 매번 소리쳐서 방향을 잡아줄 수도 있지만,
> 마구(Harness)를 채우면 말이 달리는 방향 자체를 구조적으로 제어할 수 있습니다.
> 하네스 엔지니어링은 AI에게 마구를 설계하는 일입니다.

이어서 설명할 내용:
- 2023: 단발 지시 (프롬프트 시대)
- 2025: 맥락 설계 (컨텍스트 엔지니어링 시대)
- 2026: 환경 설계 (하네스 엔지니어링 시대)

OpenAI 5개월 실험 사례 소개:
> 100만 줄 코드를 생성했지만 인간이 작성한 코드는 0줄이었다.
> 병목은 모델이 아니었다 — **환경**이었다.

질문으로 이해 확인:
> "지금 Claude Code(또는 AI)를 쓰면서 같은 실수가 반복된 적 있나요? 어떤 종류였나요?"

#### [이론 2/3] "절대 하지 마" 프롬프트가 왜 실패하는가

**튜터 진행 방식:**

다음 비유로 시작하세요:

> **"교통 규칙 vs 물리적 구조물"**
> "이 길로 가지 마세요"라는 표지판(프롬프트)은 무시될 수 있습니다.
> 하지만 중앙 분리대(시스템/구조)는 물리적으로 막습니다.
> 프롬프트는 표지판이고, 하네스는 중앙 분리대입니다.

설명할 내용:
- 에이전트는 대화 맥락이 길어질수록 초반 지시를 "희석"합니다
- `NEVER do X` 같은 부정 명령은 모델이 내부적으로 X를 먼저 떠올린 후 억제하는 구조 → 실패율이 높습니다
- Hashimoto 원칙: **"실패할 때마다 시스템을 엔지니어링하라"**

#### [이론 3/3] CLAUDE.md vs AGENTS.md

설명할 내용:
- `CLAUDE.md`: Claude Code 전용. Claude가 프로젝트 루트에서 자동으로 읽는 헌법 파일
- `AGENTS.md`: 모델 불가지론적 표준. 어떤 AI 에이전트든 읽을 수 있는 공용 표준
- 프로젝트에 하나만 쓴다면 → `CLAUDE.md` 우선. 멀티 모델 환경이면 `AGENTS.md` 병용

#### [실습 2/3] 베이스라인 측정

**튜터 진행 방식:**

다음 Node.js 스크립트를 제공하세요. 학습자가 로컬에서 실행해 기준점을 기록합니다:

```javascript
// baseline-check.js
// 실행: node baseline-check.js

const tasks = [
  {
    id: 1,
    description: "현재 프로젝트에서 AI에게 맡긴 마지막 태스크를 떠올리세요",
    questions: [
      "성공했나요? (Y/N)",
      "불필요한 파일이 변경됐나요? (Y/N)",
      "의도한 범위를 벗어난 작업이 있었나요? (Y/N)",
      "같은 지시를 몇 번 반복했나요? (숫자)"
    ]
  }
];

console.log("=== 하네스 엔지니어링 베이스라인 측정 ===\n");
console.log("아래 항목에 답하며 현재 상태를 기록하세요.");
console.log("이 기록이 Module 05에서 '개선 전/후' 비교의 기준이 됩니다.\n");

tasks.forEach(task => {
  console.log(`[태스크 ${task.id}] ${task.description}\n`);
  task.questions.forEach((q, i) => {
    console.log(`  Q${i+1}. ${q}`);
  });
  console.log();
});

console.log("결과를 baseline.md 파일로 저장해두세요.");
console.log("저장 경로 예시: .claude/baseline.md");
```

학습자의 결과를 `.claude/baseline.md`에 저장하도록 안내하세요. 이 파일은 Module 05에서 다시 사용됩니다.

#### [실습 3/3] CLAUDE.md 기본 골격 작성

**튜터 진행 방식:**

학습자의 프로젝트(React + Google Cloud Functions)에 맞는 CLAUDE.md 초안을 함께 작성합니다:

```markdown
<!-- 학습자와 함께 채워나갈 CLAUDE.md 템플릿 -->

# CLAUDE.md — 프로젝트 에이전트 헌법

## 프로젝트 컨텍스트
- 스택: React (프론트엔드) + Google Cloud Functions (Node.js 백엔드)
- [학습자가 추가: 주요 디렉토리 구조]
- [학습자가 추가: 핵심 도메인 설명]

## 4대 원칙 (Karpathy)

### 1. Think Before Coding
- 코딩 전 반드시 가정을 명시할 것
- 더 단순한 방법이 있으면 먼저 말할 것
- 혼란스러우면 숨기지 말고 질문할 것

### 2. Simplicity First
- 요청된 것만 구현할 것
- 50줄로 가능하면 200줄로 작성하지 말 것

### 3. Surgical Changes
- 내 변경이 만든 orphan 코드만 정리할 것
- 기존 dead code는 언급만 하고 건드리지 말 것

### 4. Goal-Driven Execution
- [단계 설명] → verify: [확인 방법] 형식으로 실행할 것
- 성공 기준을 먼저 정의할 것

## 절대 금지 항목 (Module 02~03에서 채워나갈 것)
- [ ] (실패 패턴 감사 결과를 여기에 추가)
```

작성 후 질문:
