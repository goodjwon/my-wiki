# 실습 1-1: 실패 패턴 감사 (Failure Audit)

## 목적
Claude Code와 작업하면서 반복되는 실패 패턴을 찾아, 각각을 "프롬프트 문제"와 "시스템 문제"로 분류합니다.
이 분류가 CLAUDE.md와 Hooks 설계의 출발점이 됩니다.

---

## Claude Code에 붙여넣을 프롬프트

```
지금 이 프로젝트에서 내가 너(AI 에이전트)와 작업하면서 반복적으로 겪었던 문제들을 분석해줘.

## 분석 대상
최근 git log를 확인해서 다음을 찾아줘:
1. 내가 revert 하거나 수정을 여러 번 반복한 커밋
2. TODO나 FIXME 주석이 남아있는 파일
3. 테스트가 없는 서비스 메서드

## 출력 형식
각 문제에 대해 다음 표 형식으로 정리해줘:

| 문제 | 반복 횟수 | 분류 | 해결 방법 |
|------|-----------|------|-----------|
| 문제 설명 | N회 | 프롬프트/시스템 | 제안 |

## 분류 기준
- **프롬프트 문제**: 지시를 더 명확히 하면 해결됨
- **시스템 문제**: 아무리 잘 말해도 구조적으로 막지 않으면 반복됨 → CLAUDE.md 규칙 또는 hooks 필요

## 추가 확인
다음 명령 실행 결과도 포함해줘:
- git log --oneline -20
- grep -rn "TODO\|FIXME\|HACK" src/ --include="*.java" | head -20
- find src -name "*.java" -path "*/service/*" | xargs grep -L "@Test" | head -10
```

---

## 예상 출력 예시

| 문제 | 반복 횟수 | 분류 | 해결 방법 |
|------|-----------|------|-----------|
| Entity를 Controller에서 직접 반환 | 4회 | 시스템 | CLAUDE.md 금지 규칙 + DTO 변환 강제 |
| migration 파일 내용 임의 수정 | 2회 | 시스템 | guard.sh로 수정 차단 |
| 테스트 없이 서비스 메서드 추가 | 6회 | 시스템 | PR 전 테스트 존재 확인 hooks |
| 변수명이 요청과 다르게 생성 | 3회 | 프롬프트 | CLAUDE.md 네이밍 컨벤션 추가 |
