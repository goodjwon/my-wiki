# Java 학습 경로 T2 — 객체지향 설계로 응용

> **이 트랙의 목표**: T1에서 익힌 객체지향을 **설계 수준**으로 끌어올린다. 디자인 패턴 8종을 "외우는" 게 아니라 *바뀌는 것과 안 바뀌는 것을 분리하는 도구*로 쓰고, JVM이 그 코드를 어떻게 돌리는지까지 이해한다.

## 🧩 핵심 개념 한눈에

| 개념 | 한 줄 정의 | 자주 하는 실수 |
|------|----------|--------------|
| 전략 패턴 | 알고리즘 군을 캡슐화해 교체 가능하게 | `if-else`로 분기해 패턴 회피 |

| GC | 도달 불가 객체 자동 회수 | "GC가 다 해주니 메모리 신경 끔" |

## 💡 이론·방법론 연결 (팁)

> 💡 **설계 — 패턴은 목적이 아니라 결과**
> 패턴을 먼저 정하고 끼워 맞추면 과설계가 된다. "무엇이 바뀌는가"를 먼저 찾으면 패턴은 따라온다.
> → 자세히: [[concept-design-patterns]] · [[lecture-object-ch9]] (유연한 설계)

> 💡 **설계 — 상속보다 합성**
> 패턴 대부분이 "상속 대신 객체를 조합"으로 귀결된다. 13장의 서브타이핑/합성 판단 기준을 같이 보라.
> → 자세히: [[lecture-object-ch13]] · [[concept-oop]]

> 💡 **실무 — 싱글톤은 스프링 빈으로**
> 4.3 싱글톤을 직접 구현하지 말고 스프링 컨테이너에 맡긴다. T4에서 이어진다.
> → 자세히: [[concept-spring-core]] (IoC/DI)

> 💡 **성능 — "왜 느린가"는 JVM에서 답이 나온다**
> 9장 GC·메모리를 알면 "객체를 너무 많이 만든다" 같은 원인을 데이터로 짚는다.
> → 자세히: [[entity-jvm]] · [[concept-db-connection-pool]] (풀로 객체 재사용)

## 🛠 미니프로젝트 — 전략 패턴으로 리팩터링

1. **전략 패턴 실전문제 (4.9)** — `if-else` 덩어리를 전략 객체로 분리. 새 전략 추가가 *기존 코드 수정 없이* 되는지 확인(OCP).
2. **메모 앱 토이프로젝트 (11.26-1)** — 콘솔 CRUD를 먼저 완성하고, 저장 방식을 전략으로 갈아끼울 수 있게 확장.

**따라하는 법**:
```text
1) 바뀌는 부분(알고리즘/정책)을 한 인터페이스로 추출
2) 구현체별 클래스 + 실패 테스트(TDD 빨강)
3) 컨텍스트가 인터페이스에만 의존하도록 (초록)
4) 분기문 제거·이름 정리 (리팩터)
5) 새 전략 1개를 "기존 코드 0줄 수정"으로 추가해 OCP 검증
```
원문 풀이: `raw/java-study/java-study-ch04-객체지향설계와패턴.md`(4.9), `java-study-ch11-부록.md`(11.26-1).
