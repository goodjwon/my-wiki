# 객체지향 프로그래밍 (OOP)

## 정의

상태와 행위를 함께 묶고 책임을 나누는 프로그래밍 패러다임. Java의 핵심 설계 철학이다.

## 4가지 원칙

| 원칙 | 핵심 | 흔한 오해 |
|------|------|----------|
| **캡슐화** | 잘못된 상태 변경을 막는 것 | getter/setter 자동 생성과 같다 |
| **상속** | is-a 관계가 분명할 때 사용 | 모든 재사용을 상속으로 해결 |
| **다형성** | 같은 메시지에 다른 응답 | 타입 캐스팅을 많이 쓰는 것 |
| **추상화** | 변경 가능성이 큰 지점을 분리 | 인터페이스를 많이 만드는 것 |

## 핵심 판단 기준

- 객체를 설계할 때 "무엇을 가지고 있는가"보다 **"무엇을 책임지는가"**를 먼저 본다
- 상속보다 **합성**이 더 안전한 경우가 많다
- 인터페이스는 **실제로 교체 가능성이 있는 지점**에만 둔다
- 생성자는 "이 객체가 유효한 상태로 시작한다"는 약속

## 실무 연결

Spring DI, 전략 패턴, 도메인 모델링, 테스트 가능한 구조는 모두 `구현이 아니라 역할에 의존한다`는 OOP 감각 위에서 작동한다.
